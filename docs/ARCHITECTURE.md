# TrailMe — Architecture & Implementation Plan

> Multi-tenant SaaS: a live Mapbox map where security guards see each other's real-time position and fading "trail" over a 5/10/15/30/60/120-minute window, so inspections don't re-cover the same ground.
>
> Produced by the `trailme-architecture` design workflow (5 research deep-dives → synthesis → 3 adversarial critiques → reconcile). This is the source-of-truth plan.

## Executive summary

TrailMe is a multi-tenant GPS-tracking SaaS for security/guard companies, built on Supabase (Postgres 15 + PostGIS + Realtime + Auth + RLS), an Expo/React Native field app (transistorsoft background geolocation), and a Next.js control-room dashboard. The core view is a live Mapbox map where every guard sees peers' live positions plus a fading "trail" and an org-wide coverage heatmap over a selectable window (5/10/15/30/60/120 min), so two guards do not re-cover the same ground.

Three concerns are decoupled so a hiccup in one never degrades the others: (1) LIVE position (ephemeral, off-database WebSocket), (2) durable BREADCRUMB history (batched, idempotent, partitioned Postgres), (3) derived COVERAGE/heatmap (client-rendered from decimated history). This finalized plan keeps the original architecture's strengths (transistorsoft SQLite queue, idempotent replay, per-site channels, server-decimated trail_window, server-authoritative gate) and folds in every sound CRITICAL and HIGH critique fix:

CRITICAL fixes applied: (1) Live Broadcast payloads are no longer trusted for identity — the receiving clients derive publisher identity from the server-stamped Realtime metadata (authenticated user_id), and live ticks are explicitly ADVISORY; guard_positions (server-written, RLS-protected) is the authoritative last-known. (2) captured_at is NEVER overwritten — true device event time is preserved (Art. 5(1)(d) accuracy); a separate partition_ts handles routing, and the idempotency key drops the rewritable timestamp. (3) The live broadcast is driven FROM the location callback, not a wall-clock timer — backgrounded peers update at movement cadence on iOS/Android, with a guard_positions polling fallback. (4) Store-submission de-risking and the in-app prominent-disclosure flow move to M2, not M6.

HIGH fixes applied: site-level (not just org-level) Realtime + trail_window authorization with live membership re-check; append-only audit log of location reads and privacy-config changes; legitimate-interest (not employee-consent) lawful basis with a per-guard disclosure-acknowledgement record and DPIA/LIA artifacts; data-minimized guard-to-guard visibility (same site + co-active shift, short trail only) defaulted safe; a decoupled long-lived device ingest token separate from the 1h interactive session; a server-side "tracking went dark" staleness watchdog; OEM battery-killer onboarding flow; offline-aware clock-in gate that never silently drops buffered breadcrumbs; client monotonic-clock guard for captured_at; GPS-accuracy filtering on ingest; and per-batch reject reporting (no silent loss). The system is architected to scale to low-hundreds of concurrent guards per org without a rewrite; the binding scaling axis is WebSocket fan-out, addressed by one CONSOLIDATED per-site broadcast (see realtime section) rather than quadratic peer-to-peer.

## Data model (Postgres + PostGIS)

Tenancy & identity:
- organizations(id uuid pk, name, stripe_customer_id nullable, created_at).
- org_settings(org_id pk fk, tracking_mode enum['shift_gated','always_on'], retention_days int check in (7,30,90), lawful_basis enum['legitimate_interest','consent'] (admin must affirm; default legitimate_interest), dpia_completed_at nullable (required non-null before always_on can be enabled — enforced by trigger), plan, seat_limit).
- sites(id uuid pk, org_id fk, name, geofence geography(Polygon,4326) nullable for auto clock-in/out).
- profiles(id = auth.users.id, display_name, color).
- memberships(id, user_id fk, org_id fk, role enum['org_admin','supervisor','guard'], site_ids uuid[], active bool default true, UNIQUE(user_id) for MVP one-org-per-user). site_ids is a fast-path hint only; sensitive reads re-check live membership.
- guard_disclosures(id, user_id, org_id, notice_version text, tracking_mode_at_accept, accepted_at) — proves what each guard was told (transparency / employer-consent-not-freely-given). Insert required before tracking starts.

Hot path (firehose):
- location_breadcrumbs PARTITION BY RANGE(partition_ts) with daily child tables. Columns: id uuidv7, org_id, guard_id, site_id, geom geography(Point,4326), captured_at timestamptz (TRUE immutable device event time — never rewritten, drives ordering/age-fade), partition_ts timestamptz (server-clamped-only-if-insane copy, used ONLY for partition routing), ingested_at, client_seq bigint (monotonic per-device counter), accuracy_m real, is_keepalive bool, is_low_confidence bool. Composite PK (id, partition_ts). Idempotency: UNIQUE(guard_id, client_seq) — deliberately EXCLUDES any server-rewritable timestamp so a clamped replay still collides. Indexes (exactly four hot + the unique): btree(org_id, captured_at DESC) INCLUDE(guard_id), BRIN(captured_at), GiST(geom), btree(guard_id, captured_at DESC). No client UPDATE/DELETE policy (immutable).
- guard_positions: thin 1-row-per-guard last-known table (org_id, guard_id, site_id, geom, heading, captured_at, accuracy_m, online_bool). Server-written, RLS-protected — this is the AUTHORITATIVE live marker source and the broadcast-relay source; live broadcast ticks are advisory hints reconciled against it.
- inspection_checkpoints(id uuidv7, org_id, guard_id, site_id, geom, label, tagged_at). Server-authoritative discrete event (realtime.send on insert).
- shifts(id, org_id, guard_id, site_id, clock_in, clock_out nullable, auto_closed bool). Gate is window-based: a breadcrumb is allowed if captured_at ∈ [clock_in, COALESCE(clock_out, clock_in + max_shift)].

Governance & accountability:
- audit_log (APPEND-ONLY): id, org_id, actor_user_id, action enum['trail_window_read','dsar_export','erasure','settings_change','channel_join_sampled'], target_guard_id nullable, params jsonb, ts. Written by SECURITY DEFINER on sensitive reads (trail_window), DSAR export, erasure, and every org_settings change. Own retention (longer than location retention). Cannot be backfilled — present from M1.
- erasure_registry: static list of every table containing guard_id, consumed by the DSAR-export and erasure RPCs so neither can miss a table.

RPCs: batch_insert_breadcrumbs (SECURITY DEFINER, builds geography, ON CONFLICT (guard_id,client_seq) DO NOTHING, upserts guard_positions, re-verifies org/site against memberships — defense-in-depth against a bad JWT claim); trail_window(p_site,p_minutes) (server-decimated, audited, re-checks live site membership of caller); create_organization; redeem_join_code; clock_in/clock_out; dsar_export (enumerates erasure_registry → portable bundle); erase_guard (hard-delete the subject's rows + checkpoints + guard_positions, tombstone guard_id so replay can't resurrect).

RLS: FORCE on every tenant table; authz.org_id()/has_role()/has_site_access(site_id) (live membership check, not JWT-only) for sensitive reads; BEFORE INSERT enforce_tracking_gate() as the window-based shift authority. Custom Access Token Hook is NULL-safe (a throw locks out all org logins) and minimal (never elevates role — covered by its own test).

## Realtime architecture

TRANSPORT — CONSOLIDATED SERVER-RELAYED BROADCAST (resolves both the fan-out billing critique and the identity-spoofing critique):

The original "every guard client-publishes its own tick to all peers" is quadratic — at 80 guards/site it is ~2133 msg/s and ~3800k msg/mo, multiples over plan fan-out caps, and (worse) lets any joined member publish a tick claiming a colleague's guard_id and location, falsifying the live coverage map that the whole product is meant to be trusted for. Both problems are fixed by NOT relaying raw client payloads peer-to-peer:

1. Each guard sends its own position upstream ONLY (to the ingest path / a light position upsert), never fanned to peers directly. Identity is the server-verified JWT subject; the client-supplied 'g' field is ignored.
2. A thin relay (relay-positions Edge Function, or a DB trigger calling realtime.send on guard_positions upsert) emits ONE consolidated 'positions' message per site per tick containing all active guards' server-stamped {guard_id, lat, lon, heading, captured_at, online}. That is ~1 msg/tick/site (~27 msg/s for a busy site, ~48k/mo) instead of quadratic. Subscribers apply the batch with a single setData. Because the relay stamps guard_id from the authenticated upsert, a guard CANNOT impersonate a peer or inject a ghost.
3. Live ticks are ADVISORY. guard_positions (server-written, RLS-protected) is authoritative last-known; a freshly-mounted dashboard seeds markers from it before the first tick and reconciles against it.

LIVE CADENCE FROM THE LOCATION CALLBACK (resolves the iOS background-suspend critique): the field app does NOT run an independent 2-5s wall-clock timer in the background (iOS suspends the process and tears down the socket within seconds; it also burns battery). Instead the position-upstream send fires inside transistorsoft's onLocation callback, which wakes the app in the background on both platforms. Background live cadence therefore equals the location cadence (distanceFilter ~20m), not a fixed 2-5s. Foreground (a guard looking at their own phone, or the control-room browser) gets smoother cadence with client-side interpolation. A guard_positions polling fallback (and optional iOS silent push / significant-location-change wake) keeps a backgrounded peer's marker recent even when the socket is dead. Expectation is documented: backgrounded peers move at movement cadence.

CHANNEL SCOPING & AUTHORIZATION (resolves intra-org cross-site leak): private channel per SITE (site:{site_id}). Realtime Authorization RLS on realtime.messages MUST assert (site_id = ANY(joiner's site_ids) AND org matches) AND a LIVE has_site_access(site_id) membership check — NOT merely the coarse org_id() claim, and not JWT site_ids alone (stale on reassignment until TTL). The channel suffix is attacker-controlled, so the parsed site_id is validated as a uuid the caller is entitled to. M3 verify explicitly tests that a guard in org A assigned to site 1 is REJECTED joining site 2 in the SAME org. A whole-org control room subscribes to N site channels; sites expected >~80 concurrent guards pre-plan shard channels.

DATA MINIMIZATION (resolves Art. 5(1)(c) critique): guard-to-guard visibility defaults to the NARROWEST operationally-sufficient scope — same site AND co-active shift, live position + the short selected-window trail only, NOT arbitrary colleague history. Org-wide and historical visibility is a supervisor/control-room capability gated by role, encoded in the Realtime Authorization policy (site membership gates join) and in trail_window RLS (a guard reads peers only for sites they are actively assigned to). This is a defaulted-safe control, decided now, not deferred.

ROSTER: Presence carries only who-is-online (guard_id, color, clocked_in, battery_level) and auto-evicts on disconnect — never coordinates. Battery level surfaces in the dashboard roster so a control room sees a guard about to go dark.

BREADCRUMB THROTTLE/BATCH (the SEPARATE durable path): persist on (≥5s elapsed) OR (≥10m moved), suppress stationary, 2-min keepalive (is_keepalive=true, excluded from the heatmap source so a parked guard makes no false hotspot). transistorsoft's SQLite queue persists each gated fix and autoSyncs batches (~5-12) to ingest with exponential backoff, authed by the 24h device token.

CAPTURED_AT INTEGRITY (resolves the clamp-vs-replay + accuracy critique): true device captured_at is NEVER rewritten (Art. 5(1)(d)). It is validated to a WIDE sane window (reject only far-future beyond small skew and absurdly-old beyond retention) so legitimately-old buffered fixes (a full offline shift) keep their real time. Partition routing uses a separate partition_ts (clamped only if insane); partitions are created several days ahead AND behind to cover the max offline buffer. The client makes captured_at monotonic per device (max(lastTs+1, gpsTs)) so an OS clock/timezone correction mid-shift can't produce out-of-order or negative-age trail points (line-gradient needs newest-LAST). Idempotency key is (guard_id, client_seq) — never a rewritable timestamp — so a clamped replay still dedups.

INGEST QUALITY & NO-SILENT-LOSS (resolves accuracy + dead-letter critiques): the Edge Function drops/flags fixes worse than ~50m accuracy and discards 'stuck' repeated last-known fixes; low-confidence points are flagged (is_low_confidence) and excluded from the heatmap, rendered visually distinct in trails. It returns a PER-BATCH result (accepted count + rejected rows with reason); gate-rejected rows (clock-in not yet synced) are kept for bounded retry/reconciliation rather than dropped, unparseable rows go to a dead_letter table, and a rejected-count metric is surfaced so loss is observable. The client deletes from its local queue only after a 200 that confirms persistence of the kept rows.

OFFLINE GATE RECONCILIATION (resolves server-gate-vs-offline-buffer silent-loss critique): clock-in is optimistic and locally cached so the SDK keeps recording; the gate is window-based on captured_at, and a late-syncing clock-in reconciles buffered breadcrumbs into the shift window rather than the trigger hard-rejecting them as 'tracking_off_shift'. Geofence-assisted auto clock-in/out (site geofence enter/exit) backstops a forgotten clock-in; pg_cron auto-closes forgotten clock-outs after a max-shift window so shift-gated never tracks indefinitely.

TIME-WINDOW TRAIL + HEATMAP: on mount/window change, ONE trail_window(p_site,p_minutes) RPC returns SERVER-DECIMATED data (time-bucketed ~1pt/3s + ST_Simplify), capped to a few thousand points (mobile capped much tighter), audited, and live-membership-checked. The client holds the 120-min buffer; a single ~1Hz client clock re-derives all three Mapbox GeoJSON sources with zero refetch on shrink: (1) breadcrumbs heatmap weight age_norm=(now−t)/windowSec, keepalive + low-confidence excluded; (2) per-guard LineString trails newest-LAST, lineMetrics:true, window enforced by TRIMMING coordinates (line-gradient ignores GL filters), one ShapeSource/LineLayer per active guard for distinct hue; (3) live markers via throttled setData (~1Hz, one FeatureCollection) + feature-state for cheap color/heading/online flicker. Growing the window may refetch; shrinking is a pure client trim. Heatmap is org-wide MONOCHROME recency density (not per-guard tint), heatmap-color starting at rgba(...,0).

WRITE-VOLUME MATH: 500 guards × 1 persist/5s ≈ 100 rows/s (~half with distance-gating) — ~1-3% of a small Postgres instance. Postgres is never the live bottleneck; consolidated-broadcast fan-out is bounded per site.

## Security & privacy (GDPR)

LAWFUL BASIS — legitimate interest, NOT employee consent. Under EDPB guidance employee consent is generally not freely given (employer-employee power imbalance), so the relied-upon basis is GDPR Art. 6(1)(f) legitimate interest (guard safety/coordination, lone-worker protection, preventing duplicate coverage), backed by a documented balancing test (LIA in docs/GDPR.md). Always-on tracking of an identified person is high-risk and DPIA-mandatory in Norway: org_settings.dpia_completed_at must be non-null before an org can enable always_on (enforced by trigger). org_settings.lawful_basis is an admin-affirmed field; shift_gated is the recommended default. Datatilsynet's specific positions on employee GPS tracking are captured in the runbook.

TRANSPARENCY & DISCLOSURE-ACKNOWLEDGEMENT. Before any tracking starts, each guard must accept an in-app prominent-disclosure screen (shown BEFORE the OS permission prompt — also a store-approval requirement). The acceptance is recorded in guard_disclosures(notice_version, tracking_mode_at_accept, accepted_at) so the org can prove exactly what each guard was told. iOS keeps the blue location indicator visible; Android keeps the sticky foreground-service notification — never suppressed.

RLS PATTERN — defense in depth, site-granular.
- FORCE RLS on every tenant table. The hot path uses the constant-time authz.org_id() JWT claim for org isolation, but sensitive cross-guard reads (trail_window, exports) ALSO re-verify org/site/role against the memberships table inside SECURITY DEFINER RPCs — a forged/buggy JWT claim cannot read another tenant's data (a per-RPC invariant, tested, not just a pen-test line).
- Site-level authorization: has_site_access(site_id) does a LIVE membership lookup (not JWT site_ids alone). It gates Realtime channel join (realtime.messages policy asserts site_id ∈ caller's live sites AND org match) and trail_window. This closes both intra-org cross-site leakage and the stale-JWT reassignment window (a guard moved off a site loses access immediately, not after the 1h TTL).
- The Custom Access Token Hook is minimal and NULL-safe (a throwing hook locks out all org logins) and is covered by a test asserting it never elevates role.
- Live Broadcast carries no authority: identity is server-stamped from guard_positions upserts; client 'g' is ignored. Live ticks are advisory; guard_positions is the RLS-protected source of truth.

CONSENT/TRACKING-CONSENT (org-configurable gate). The tracking gate (shift_gated vs always_on) and retention (7/30/90) are SERVER-side authorities: enforce_tracking_gate() is window-based on captured_at ∈ [clock_in, COALESCE(clock_out, clock_in+max_shift)] so a forgotten clock-out (auto-closed nightly by pg_cron) or an offline-buffered late clock-in can't track indefinitely or silently drop legitimate data. The app additionally refuses to start GPS when the gate says no (battery + privacy), but the DB is the authority.

ACCOUNTABILITY — audit log (Art. 5(2)). An append-only audit_log records who read which guard's trail (trail_window), every DSAR export, every erasure, and every privacy-material config change (tracking_mode, retention_days, lawful_basis). Present from M1 (impossible to backfill). Channel-join is sampled from Realtime logs. This supports DSAR responses and abuse detection (e.g. a supervisor stalking one guard).

RETENTION & STORAGE LIMITATION (Art. 5(1)(e)). pg_cron: nightly create-ahead/behind partitions; DROP partitions past the max org retention; per-org soft-purge DELETE for 7/30-day tenants sharing a 90-day partition (autovacuum tuned for the resulting bloat); separate longer retention for audit_log. The high-value identifiable firehose gets the shortest retention; the derived coverage layer is computed from already-decimated/grid-snapped data (lower re-identification risk).

DSAR & ERASURE (Art. 15/17), resolved BEFORE first real customer data. A guard's data spans partitioned breadcrumbs, guard_positions, inspection_checkpoints, shifts, presence, audit_log, and disclosures. dsar_export enumerates erasure_registry (a maintained list of every table with guard_id) into one portable bundle so nothing is missed. A continuous one-person GPS trail is re-identifiable on its own (it ends at their home/locker), so "anonymize guard_id" is pseudonymization, not anonymization — erase_guard prefers HARD-DELETE of the subject's rows and their checkpoints, and tombstones the guard_id so idempotent replay (keyed on guard_id, client_seq) cannot resurrect erased rows, and clears guard_positions/Presence.

SECURITY OF PROCESSING & POSTURE (Art. 32/33/34/28). docs/GDPR.md documents: TLS in transit (Supabase) and at-rest encryption; a breach-detection + 72-hour notification runbook (Art. 33/34); Supabase as processor with a DPA and sub-processor posture; and a pinned EU data-residency guarantee for the Supabase project AND the Realtime relay region (Norwegian customers will ask). The 1h interactive access-token TTL + auth.admin.signOut on offboarding + live has_role()/has_site_access() checks revoke a removed guard's reads/realtime within the TTL; the separate device ingest token is revocable independently.

## Changes applied from adversarial critique

- LIVE IDENTITY (critical, privacy): Dropped client peer-to-peer broadcast as authority. Each guard sends position upstream only; a relay emits ONE consolidated per-site message with server-stamped guard_id. Receiving clients ignore the client 'g' field; guard_positions (RLS-protected) is authoritative; live ticks are advisory. Kills both impersonation/ghost-injection and the quadratic fan-out billing problem at once.
- FAN-OUT BILLING (critical, scaling): Consolidated one-broadcast-per-site-per-tick (~27 msg/s) replaces quadratic peer-to-peer (~2133 msg/s, 4x over plan caps). Documented one web map per session and grow=client-trim to bound Mapbox per-load and trail_window refetch.
- CAPTURED_AT INTEGRITY (critical, privacy + mobile): True device captured_at is NEVER rewritten (Art. 5(1)(d) accuracy); a separate partition_ts handles routing (clamp only if insane); partitions created ahead AND behind for the offline buffer; legitimately-old buffered fixes keep their real time. Idempotency key changed to (guard_id, client_seq) — no rewritable timestamp — so clamped replays still dedup. Client makes captured_at monotonic to survive OS clock/timezone corrections.
- iOS BACKGROUND LIVE (critical, mobile): Live send is driven FROM transistorsoft's onLocation callback, not a wall-clock 2-5s background timer (iOS suspends + tears down the socket). Background cadence = movement cadence; guard_positions polling (and optional silent push) is the fallback; expectation documented.
- STORE DE-RISKING (critical, mobile): Moved early TestFlight/internal-track submission with background location, the in-app prominent-disclosure flow, and the Play Permissions Declaration text to M2 (when GPS first works), not M6 — surfacing review objections while the cost to change is low.
- SITE-LEVEL AUTHZ (high, privacy): Realtime channel join and trail_window now require has_site_access(site_id) — a LIVE membership check, not the coarse org_id() claim and not JWT site_ids alone. ## Appendix A — Critique findings (traceability)

### Lens: scaling-cost

*Broadcast fanout billing breaks live transport*

- **[CRITICAL] Broadcast bills per-recipient fan-out: peer-to-peer is quadratic in guards/site, ~2133 msg/s and 3800/mo per 80-guard site, 4x over Pro's 500 cap.**
  - Fix: One consolidated broadcast per site per tick: ~27 msg/s, 48/mo. Shard tenants per project.
- **[HIGH] Unmodeled: peak connections, Mapbox web per-load, trail_window refetch on mount/grow.**
  - Fix: COGS-per-org model; one web map per session; cache trail_window, grow = client trim.
- **[MEDIUM] 1Hz heatmap restyle risks mobile jank; egress and soft-purge bloat unbudgeted.**
  - Fix: Cap mobile points; per-tier partition-drop; gzip egress.

### Lens: privacy-gdpr

*The architecture is privacy-aware in intent (server-side gate, RLS, per-tenant retention, GDPR.md) but has real, exploitable holes in the live layer and several GDPR gaps that are structural, not cosmetic. The most serious: the client-published Broadcast path has NO server-side authorization of payload contents, so any authenticated guard can spoof a peer's identity and location into the live map; RLS on realtime.messages only gates channel join/read, not what payload a member may publish. Separately, the captured_at clamping rule directly contradicts the offline-replay requirement and will silently falsify the timestamp (and thus the location history) of legitimately-buffered fixes — an accuracy/integrity violation. Audit logging of who-viewed-whose-location is entirely absent, which is hard to defend for person-level tracking data under Norwegian/EU rules. The retention and tenant-isolation design on the persist path is mostly sound, but consent/lawful-basis handling, DSAR scope, and erasure-vs-retention are under-specified in ways that will block a defensible launch.*

- **[CRITICAL] Live Broadcast payloads are unauthenticated and unvalidated server-side. The design relies on 'Realtime Authorization RLS on realtime.messages at join' for the live layer, but client-published Broadcast (ch.send({type:'broadcast', event:'pos', payload:{g,lat,lon,...}})) is a pure WebSocket relay — RLS only controls whether a client may JOIN/read a channel, not what payload it may emit once joined. Any guard who is a legitimate member of site:X can publish a tick claiming to be a DIFFERENT guard's id (g) at an arbitrary lat/lon. There is no signature, no binding of the broadcast 'g' to the publisher's JWT subject. This lets a guard (a) impersonate a colleague's live position, (b) inject ghost guards, (c) make the live coverage map lie during an inspection — directly defeating the core 'don't re-cover ground' use case and creating a falsifiable record. This is the single biggest hole in the design.**
  - Fix: Do not trust client-published coordinates for anything authoritative. Two-part fix: (1) Bind identity: on the receiving clients, ignore the payload 'g' field entirely and derive the publisher identity from the Realtime presence/ack metadata that the server attaches to the message (the authenticated user_id), or drop client Broadcast for position and instead publish live ticks via a thin Edge Function / realtime.send that stamps the server-verified guard_id. (2) If keeping client Broadcast for latency, treat it as a soft hint only and reconcile against guard_positions (server-written, RLS-protected) which is the authoritative last-known; never let a client tick move a marker attributed to a guard_id the publisher does not own. Add a Realtime Authorization policy that also restricts the broadcast event, and document that live ticks are advisory, not evidentiary.
- **[CRITICAL] captured_at clamping contradicts offline replay and corrupts location-time accuracy. The Edge Function 'clamps device captured_at to within ±a few minutes of ingested_at' to protect partition routing. But the SDK's durable SQLite queue is explicitly designed to replay fixes recorded during long outages (a full off-shift, stairwells, basements) on reconnect — those fixes are LEGITIMATELY tens of minutes to hours old. Clamping them to ~now writes the wrong timestamp, which (a) mis-orders the trail, (b) places a guard at a location at a time they were not there, and (c) violates GDPR Art. 5(1)(d) accuracy for personal data. The two requirements are in direct conflict and the design treats clamping as unconditionally safe.**
  - Fix: Separate the two timestamps and never overwrite the real one. Keep device captured_at as the authoritative event time (validated to a wide sane window, e.g. not in the future beyond small clock skew, not absurdly old like >retention) and store ingested_at separately. Solve partition routing by (1) creating partitions a few days AHEAD and keeping a 'catch-all'/default partition or pre-creating enough past partitions to cover the max offline window, or (2) routing on a clamped-for-routing-only column while persisting true captured_at unchanged. Reject (don't silently clamp) far-future timestamps; accept legitimately-old buffered fixes into their correct daily partition. Clamp must never be allowed to rewrite the recorded time of a real fix.
- **[HIGH] No access/audit logging of location reads. For person-level GPS tracking in the EU/Norway, who viewed whose location (and when) is itself a controllership/accountability obligation (GDPR Art. 5(2) accountability; supports DSAR and abuse detection — e.g. a supervisor stalking a specific guard). The design logs nothing on the read path: trail_window RPC, live channel subscriptions, and dashboard views leave no trail. There is also no record of which admin changed tracking_mode or retention (a privacy-material config change).**
  - Fix: Add an append-only audit table (org_id, actor_user_id, action, target_guard_id, params, ts) written by SECURITY DEFINER on sensitive reads/changes: trail_window calls (who pulled which site/window/guard), exports (DSAR RPC), erasure RPC, and every org_settings change (tracking_mode, retention_days). Keep it on its own retention. Channel-join events can be sampled via Realtime logs. This is cheap to add now and effectively impossible to backfill later.
- **[HIGH] Lawful basis and consent are asserted but not engineered, and 'consent' is the wrong basis for an employer. The plan mentions 'always-on mode needs an explicit lawful basis + disclosure' but the data model has no consent/basis record, no per-guard acknowledgement timestamp, and no notion that consent from an employee is generally NOT freely given under EDPB guidance (employer-employee power imbalance) — so the real basis is legitimate interest (Art. 6(1)(f)) requiring a documented balancing test (LIA/DPIA), with always-on tracking of a person being high-risk and likely DPIA-mandatory in Norway. There is no DPIA artifact, no record of the disclosure each guard saw/accepted, and no per-org toggle to capture the chosen lawful basis.**
  - Fix: (1) Add a consent/disclosure-acknowledgement record per guard per org (version of notice, accepted_at, tracking_mode in effect) so you can prove what each guard was told. (2) Make GDPR.md include a DPIA template and an LIA for always-on mode, and surface 'lawful basis' as an org_settings field the admin must affirm. (3) Treat employee consent as not relied-upon; default messaging to legitimate-interest + transparency, and ensure shift-gated is the recommended default. (4) Add Datatilsynet-specific guidance (Norway has had specific positions on employee GPS tracking) to the runbook.
- **[HIGH] Peer guards see org/site-wide live positions and full trails — data minimization not addressed. The design states guards see org-wide trails ('current design org-wide') and every guard on a site subscribes to every other guard's live ticks and trail. For the stated use case (don't re-cover ground on the SAME site/inspection) guards arguably need same-site, same-active-shift visibility — not org-wide, and not historical trails of colleagues beyond the operational window. Org-wide peer-to-peer location visibility is a data-minimization (Art. 5(1)(c)) and proportionality problem distinct from the supervisor view, and it is currently an open product question rather than a defaulted-safe control.**
  - Fix: Default guard-to-guard visibility to the narrowest operationally-sufficient scope: same site AND co-active shift only, live position + short trail only (the selected coverage window), not arbitrary history. Keep org-wide visibility a supervisor/control-room capability gated by role, not a guard capability. Encode this in the Realtime Authorization policy (site membership, not org membership, gates channel join) and in trail_window RLS (a guard may read peers only for sites they're actively assigned to). Make this a defaulted decision, not deferred to M4-M6.
- **[HIGH] Realtime channel authorization granularity may leak across sites within an org, and channel name is client-supplied. Private channel is site:{site_id} and join is gated by 'app_metadata' membership. memberships carries site_ids[] (an array) and the Custom Access Token Hook stamps site_ids into the JWT. If the Realtime Authorization policy checks only org membership (org_id claim) rather than that the specific site_id is in the joiner's site_ids[], any guard in the org can subscribe to ANY site's live channel in that org — reading colleagues' positions at sites they're not assigned to. The design's RLS examples emphasize the constant-time org_id() claim, which is exactly the too-coarse check that would cause this.**
  - Fix: The Realtime Authorization RLS on realtime.messages for site:{id} MUST assert (id = ANY(site_ids from JWT)) AND org match, not just org match. Add an explicit test in M3 verify: a guard in org A assigned to site 1 is REJECTED joining site 2's channel in the same org (not only cross-org rejection). Also validate the parsed site_id is a uuid the joiner is entitled to, since the channel suffix is attacker-controlled.
- **[MEDIUM] Stale JWT claims weaken site-revocation and reassignment, not just org-removal. The design correctly flags ~1h TTL + admin.signOut for offboarding, but site_ids[] is baked into the JWT for up to the TTL. If a guard is reassigned off a site (or a site is offboarded), they retain read access to that site's live channel and (via trail_window if it trusts the JWT site_ids) its trails until refresh. For location data this is a window of unauthorized access to a specific person's movements.**
  - Fix: For the live channel and trail_window, do not trust JWT site_ids alone for sensitive reads — add a live membership check (has_site_access(site_id) querying memberships) in the Realtime Authorization policy and in trail_window, mirroring the design's own 'has_role() live-check for admin'. Keep JWT site_ids only as a fast-path hint. Optionally shorten TTL further for the field app or push a forced re-auth on reassignment.
- **[MEDIUM] DSAR export and erasure scope is under-specified and the erasure-vs-retention conflict is unresolved at the data layer. M6 lists a DSAR export RPC and an erasure RPC, and flags hard-delete vs anonymize as a 'product/legal decision', but: (1) a guard's personal data is spread across partitioned breadcrumbs, guard_positions, inspection_checkpoints, shifts, presence/audit, and the live layer — the export must cover all of them or it's incomplete (Art. 15). (2) Anonymizing guard_id in breadcrumbs does not anonymize a trail: a continuous GPS path of one person on one shift is re-identifiable on its own (it ends at their home/locker). 'Anonymize guard_id' is likely pseudonymization, not anonymization, so it stays in scope of GDPR. (3) Idempotent replay keyed on (guard_id, client_seq, captured_at) means erasing/anonymizing guard_id can break dedup or leave orphans.**
  - Fix: (1) Define DSAR export as a single RPC that enumerates every table containing guard_id (maintain a registry) and outputs a portable bundle. (2) Treat trail data as re-identifiable; for erasure prefer hard-delete of the data subject's rows (and checkpoints they created) rather than guard_id anonymization, unless you can show the residual trail is truly non-identifying (usually you can't). (3) Decouple the dedup key from identity where possible, or accept that erasure tombstones a guard_id so replays can't resurrect it. Resolve this BEFORE first real customer data, not at M6.
- **[MEDIUM] guard_positions and breadcrumb writes are server-authorized only by the gate trigger and RLS — but the trigger's authority depends on shift state that the client can influence. enforce_tracking_gate() is the server-side authority for shift-gated mode, checking is_tracking_allowed(). Clock-in/out is driven by client RPCs (M1). A guard who clocks in, walks off-duty without clocking out, keeps generating 'authorized' breadcrumbs — but more importantly the privacy promise of shift-gated ('we only track you while clocked in') is only as strong as the clock-out being reliable. If the app crashes or loses connectivity while clocked in, the SDK keeps buffering fixes that the gate will later ACCEPT on replay because the shift row still shows open. This can record location well past the intended shift, undermining the core privacy control.**
  - Fix: Add server-side shift hygiene: auto-close shifts after a max duration / inactivity (pg_cron), and have enforce_tracking_gate() reject breadcrumbs whose captured_at falls outside the shift's [clock_in, clock_out] (or [clock_in, clock_in+max]) window — not merely 'a shift is open now'. This makes the privacy boundary about WHEN the fix occurred, matching the offline-replay reality, and prevents indefinite tracking from a forgotten clock-out.
- **[MEDIUM] Custom Access Token Hook is a single point of org-wide lockout AND a single point of privilege escalation. The plan notes the NULL-safety lockout risk, but the more security-relevant point is that this hook stamps org_id/role/site_ids that ALL hot-path RLS trusts as constant-time truth. A bug or injection that stamps the wrong org_id/role grants cross-tenant read of location data. There is no mention of validating the hook's inputs or of a defense-in-depth check that the claimed org_id actually matches a memberships row at sensitive read time.**
  - Fix: Defense in depth: for the most sensitive reads (cross-guard trail_window, exports), don't rely on the JWT claim alone — re-verify org/site/role against memberships in the SECURITY DEFINER RPC. Add tests that a tampered/forged claim cannot read another org's breadcrumbs (the design already wants this in M6; make it a per-RPC invariant, not just a pen-test line item). Keep the hook minimal, NULL-safe, and covered by its own test that asserts it never elevates role.
- **[LOW] Live coordinates transit a relay and are 'dropped' on outage, but there's no mention of TLS/at-rest guarantees being explicitly in scope, nor of minimizing precision. All ticks are full-precision lat/lon of an identified person. For some control-room uses, full precision history is necessary; for others (e.g. heatmap), reduced precision suffices, and storing reduced-precision derived data lowers breach impact.**
  - Fix: Document transport (Supabase = TLS) and at-rest encryption as part of GDPR.md security measures (Art. 32). Consider storing the heatmap/coverage layer from already-decimated/grid-snapped data rather than full-precision points, so the high-value identifiable firehose has the shortest retention and the derived coverage layer (lower re-identification risk) can be kept longer if needed.
  - _Missing:_ A DPIA (Data Protection Impact Assessment) — for systematic, continuous location monitoring of identifiable employees this is very likely mandatory in Norway/EU; the plan has GDPR.md but no DPIA artifact and no trigger to require always-on orgs to complete one.; Any audit log of location-data READS (who viewed which guard's trail/live position) and of privacy-config changes (tracking_mode, retention_days) — required for accountability and abuse detection.; Server-side validation/binding of live Broadcast payloads to the authenticated publisher — identity spoofing on the live layer is currently unguarded.; A consent/disclosure-acknowledgement record per guard (what notice version they saw, when) — needed to prove transparency and to handle the employer-consent-not-freely-given problem.; Resolution of the captured_at-clamp vs offline-replay contradiction; as written, legitimate buffered fixes get falsified timestamps.; Site-level (not just org-level) authorization checks on the Realtime channel join and on trail_window — current emphasis on the org_id() claim risks intra-org cross-site leakage.; Breach detection/notification runbook (Art. 33/34 72-hour clock) — GDPR.md mentions DSAR/erasure but not breach response.; A data-processing agreement / sub-processor posture (Supabase as processor, region/residency of EU data, Realtime relay region) — relevant since this is multi-tenant SaaS selling to other controllers.; Erasure semantics for the live/Presence/Broadcast layer and for the idempotency dedup key — erasing a guard must also handle their last-known position and not be defeated by replay resurrection.; Explicit data-residency guarantee (EU region for Supabase project + Realtime) — Norwegian customers will ask; nothing in the plan pins region.

### Lens: mobile-reliability

*The persist-path architecture (transistorsoft SQLite queue, gated batches, idempotent replay, captured_at clamping) is genuinely well-reasoned and will survive the field — this is the strongest part of the design. But the design materially under-specifies the parts of mobile reality that actually generate the support tickets: iOS background-execution constraints on the *live* broadcast path, OEM battery-killer mitigation beyond "the SDK handles it," guard UX for clock-in/out failure modes, and a concrete app-store launch plan. Several stated mechanisms (2-5s live broadcast in background, JWT refresh while pocketed, monotonic captured_at ordering) will not behave as written on real iOS/Android. Sound foundation, but it will fail in the field at the live layer and at store submission unless the gaps below are closed.*

- **[CRITICAL] The live Broadcast path assumes the app can publish a WebSocket tick every 2-5s while backgrounded/screen-off. This is the central product feature (peers see each other move live), and it will NOT work on iOS and is unreliable on Android. iOS suspends the app process within seconds of backgrounding; a background-location app gets CLLocationManager callbacks (which is how transistorsoft keeps the persist path alive) but does NOT get a live network socket or a reliable timer to send a 2-5s broadcast. The WebSocket connection is torn down on suspend. So 'every guard sees every other guard's real-time position' degrades to 'updates only when a location callback happens to wake the app AND the socket happens to be reconnected' — i.e. the live map silently stops moving for a pocketed phone, which is the exact field scenario (phone in pocket, screen off, walking a patrol). The design treats live and persist as symmetric background paths; they are not.**
  - Fix: Drive the live broadcast FROM the location callback, not from an independent 2-5s timer. transistorsoft's onLocation fires in the background on both platforms — publish the broadcast tick inside that callback (re-establishing/using the realtime socket opportunistically), accepting that background live cadence equals the location cadence (distanceFilter ~20m), not a wall-clock 2-5s. Set expectations: backgrounded peers update on movement, not on a fixed timer. For the control room (foreground browser) cadence is fine; for peer guards looking at their own phone, the map is foreground so it's fine — but document that a backgrounded peer's marker moves at movement cadence. Consider a server-side 'derive live from latest breadcrumb' fallback so a guard whose socket is dead still shows a recent position via guard_positions polling. Also evaluate iOS silent push / significant-location-change as a wake mechanism for stale rooms.
- **[CRITICAL] App-store review risk is acknowledged in the risks list but there is NO concrete plan, and the design makes a choice that maximizes rejection probability. Apple rejects background-location apps that don't make background tracking core to *user-facing* functionality, and increasingly scrutinizes 'tracking employees' use cases. The guard FIELD app shows peers on a map (user-facing benefit), which helps — but always-on mode (org-configurable) with the screen off and no active user benefit at that moment is exactly what gets flagged. Google Play requires a prominent-disclosure video AND a Permissions Declaration form for ACCESS_BACKGROUND_LOCATION + foreground-service-location, and holds first submission. The roadmap defers ALL of this to M6 ('store readiness'), meaning you discover rejection AFTER building everything. This is schedule risk that can add weeks-to-months and is discovered last.**
  - Fix: Move store-submission de-risking to M2 (the moment background GPS works), not M6. Submit an early TestFlight/internal-track build with background location to surface review objections while the cost of changing the disclosure UX is low. Write the prominent-disclosure flow and Permissions Declaration justification text in M2. Make the in-app prominent disclosure (the consent screen explaining WHY background location is used, shown BEFORE the OS prompt) a first-class M2 deliverable, not an M6 asset. Prepare the Apple-facing justification framing background tracking as a guard-safety/coordination feature the guard themselves benefits from (peer awareness, lone-worker safety), not 'employee monitoring.' Budget review iteration time explicitly in the launch plan.
- **[HIGH] JWT refresh while the phone is pocketed and offline for a long shift is hand-waved. The design says 'onAuthorization -> refreshSession -> setConfig' handles expiry mid-outage, and separately sets access-token TTL to ~1h for fast revocation. These two requirements collide in the field: a guard works a 10-hour shift in a building with patchy signal. The refresh token can refresh while online, but if the device is offline across the 1h access-token expiry, the buffered batches cannot flush until reconnect AND a successful refresh — and Supabase refresh tokens can rotate/expire, and a refresh requires network. Worse, if the app process was killed and only the background-location service is alive, can it even run the supabase-js refresh flow? transistorsoft's HTTP autoSync uses its own native HTTP stack with a static auth header, not supabase-js — so the 'onAuthorization -> refreshSession' dance happens in JS, which may not be running when the native sync fires. Result: a long offline shift flushes with a stale/expired token and the Edge Function 401s the whole buffer.**
  - Fix: Decouple ingest auth from the short-lived user access token. Either (a) have the Edge Function accept the refresh token / a long-lived device-scoped token and mint its own short token server-side, or (b) issue a dedicated, longer-lived 'device ingest token' (signed JWT with org_id/guard_id/site claims, e.g. 24h TTL) used ONLY by transistorsoft's native HTTP sync, separate from the supabase-js session used for reads/realtime. Keep the 1h TTL on the interactive session for revocation, but don't let it gate the durable-buffer flush. Verify the native sync's auth-header refresh path actually works when JS is not running (transistorsoft's setConfig requires JS); if it doesn't, design (b) is mandatory. Test: 1h+ airplane-mode with app process killed, then reconnect — buffer must flush.
- **[HIGH] captured_at is device-supplied, drives partition routing AND idempotency key AND trail ordering — but device clocks drift, can be set manually, and (critically) jump when the user changes timezone or the OS corrects NTP mid-shift. The design clamps captured_at to +/- minutes of ingested_at at the Edge Function to protect partition routing. But clamping breaks the OTHER two uses: (1) the idempotency key is UNIQUE(guard_id, client_seq, captured_at) — if the Edge Function rewrites captured_at on clamp, a replayed batch that gets clamped to a DIFFERENT ingested_at-relative value will NOT collide and you get DUPLICATES, defeating the whole ON CONFLICT scheme. (2) Trail ordering and age-fade depend on captured_at being monotonic per guard; a backward clock correction mid-walk produces out-of-order or negative-age points that break line-gradient (which needs newest-LAST) and produce a tangled trail. The offline buffer makes this worse: buffered points captured over an hour all arrive at once and clamping near-future ones is fine, but clamping legitimately-old buffered points (captured 50 min ago, ingested now) would corrupt them if the clamp window is too tight.**
  - Fix: Separate the three concerns. (1) Idempotency key must NOT include a value the server rewrites — use UNIQUE(guard_id, client_seq) alone (client_seq is a monotonic per-device counter that already uniquely identifies a fix), or store BOTH the raw device captured_at (immutable, for the key) and a separate clamped/server-validated partition_ts. (2) Clamp window must be wide enough to accommodate the legitimate offline buffer (a guard can buffer for a full shift) — clamp only against absurd values (far future, pre-epoch), not against 'recent', or you'll truncate genuine 50-minute-old buffered points. Route partitions by ingested_at or a clamped-only-if-insane device ts, but keep raw device ts for ordering. (3) Make captured_at monotonic per-device on the CLIENT (use a monotonic clock / max(lastTs+1, gpsTs)) so a backward OS clock correction can't produce out-of-order trail points. Test with a device whose clock you manually roll backward mid-walk.
- **[HIGH] OEM battery-killer mitigation is reduced to 'transistorsoft handles Samsung/Xiaomi/Huawei via foreground service.' That is necessary but NOT sufficient and will produce the #1 field complaint: 'the app stopped tracking and we didn't know.' Aggressive OEMs (Xiaomi MIUI, Huawei, Oppo/Vivo, Samsung's 'Deep Sleep'/'Sleeping apps') will kill or freeze even a foreground-service app unless the USER manually exempts it from battery optimization and locks it in the recents tray. transistorsoft cannot do this silently — it requires guiding the user through OEM-specific settings screens (REQUEST_IGNORE_BATTERY_OPTIMIZATIONS, plus per-OEM autostart/protected-apps screens that have no API). The design has no detection-of-being-killed and no guard/supervisor alerting for 'this guard's tracking went dark.' For a security product where coverage gaps are the whole point, a silently-dead tracker is a safety and liability failure, not just a UX annoyance.**
  - Fix: Add three things. (1) An onboarding 'tracking health' setup flow that detects the OEM and deep-links the user to the right battery-exemption/autostart screens (libraries like react-native-device-info + known OEM intents; transistorsoft also exposes requestIgnoreBatteryOptimizations and deviceSettings helpers — use them). (2) A server-side 'staleness watchdog': if a clocked-in guard's guard_positions/last breadcrumb hasn't updated within N minutes, raise a dashboard alert and optionally push-notify the guard ('tracking may have stopped, open the app'). This is also a real safety feature (lone-worker down detection). (3) Surface a persistent in-app 'tracking OK / tracking degraded' indicator the guard can glance at. Treat 'silently dead tracker' as a P0 field failure mode with explicit detection, not an SDK assumption.
- **[HIGH] Clock-in/out guard UX and its failure modes are essentially unspecified, yet in shift-gated mode this gate controls whether ANY tracking happens — and guards are non-technical, often gloved, in the dark, in a hurry. Failure modes the design ignores: (a) Guard forgets to clock in -> entire patrol is invisible to peers and uncovered in the breadcrumb record (coverage gap = the product's core failure). (b) Guard clocks in but the clock-in RPC fails offline (no signal at shift start in a basement parking garage) -> the BEFORE INSERT enforce_tracking_gate trigger then REJECTS all their breadcrumbs server-side as 'tracking_off_shift', and the SDK's ingest gets a hard rejection it can't fix by retrying -> data silently lost for the whole shift even though the phone was recording. (c) Guard forgets to clock out -> always-recording, battery drain + GDPR over-collection. The trigger-as-authority design means a client/server disagreement about shift state = silent data loss, and the offline-first buffer collides badly with a server-authoritative gate.**
  - Fix: Decide how the gate behaves under offline clock-in. Option: make clock-in optimistic and locally cached so the SDK keeps recording, and have the Edge Function gate be GRACE-aware — accept breadcrumbs whose captured_at falls within a shift that gets reconciled when the clock-in RPC eventually syncs, rather than hard-rejecting. Never let a server-gate rejection silently drop a buffered breadcrumb the device legitimately recorded; at minimum return a distinct response so the client can surface 'X points rejected — were you clocked in?'. Add geofence-assisted auto clock-in/out (enter/exit site geofence) as a forgot-to-clock-in backstop. Add a loud, unmissable on-screen shift state (big 'ON SHIFT / OFF SHIFT' banner) and a missed-clock-out reminder push. Treat clock-in as a reliability-critical path with offline correctness, not a simple RPC.
- **[MEDIUM] Battery drain itself — the thing guards and their employers complain loudest about — has no budget, no measurement plan, and a configuration that maximizes it. distanceFilter ~20m with motion-detection is reasonable, but a guard walking a building slowly (sub-20m moves, constant low-speed motion) keeps GPS + accelerometer hot for an entire 8-12h shift, and the live-broadcast path (if implemented as a 2-5s timer per the design) keeps the radio/CPU awake continuously. On older/cheap Android devices (common for issued guard phones) this can drain the battery before shift end, which directly defeats the product. There is no stated target (e.g. '<X% per hour'), no plan to measure it on a representative low-end device, and always-on mode doubles the exposure.**
  - Fix: Set an explicit battery budget (e.g. <8-10%/hr on a mid-range Android) and measure it on a real low-end issued-device-class phone as an M2 exit criterion. Tune desiredAccuracy/distanceFilter/elasticity per shift-mode; consider lowering accuracy when stationary. Don't run a wall-clock live-broadcast timer in the background (see live-path issue) — riding the location callback also cuts battery. Recommend (or require) that issued phones disable battery optimization for the app AND that orgs provision power banks for long shifts. Surface battery level in the dashboard roster so a control room sees a guard about to go dark from a dead battery.
- **[MEDIUM] Signal loss in stairwells/basements is named but the offline-buffer answer is incomplete for the LIVE path and for GPS *quality*, not just connectivity. Two distinct problems are conflated: (1) no NETWORK (handled by the SQLite buffer — fine). (2) no GPS FIX (deep basement, steel/concrete) — the SDK simply produces no fixes or wildly inaccurate ones (accuracy 50-2000m, or last-known stale fixes the OS keeps returning). The breadcrumb table will then either have a gap (acceptable) or, worse, ingest garbage points jumping across the map and polluting the heatmap/coverage view with false coverage or false gaps. The design has no accuracy filtering on ingest. A guard who is actually in the basement may show as 'covered' a block away, or two guards may appear to overlap when they don't — directly defeating the 'don't re-cover the same ground' use case.**
  - Fix: Filter on horizontal accuracy at capture and/or ingest: drop or flag fixes with accuracy worse than a threshold (e.g. >50m), and never feed low-accuracy points into the coverage heatmap (similar to how keepalive points are excluded). Detect and discard 'stuck' identical fixes (OS returning the same stale last-known location repeatedly). Consider rendering low-confidence segments visually distinct (dashed/faded) so a supervisor doesn't trust a jittery basement trail as real coverage. Indoor/vertical (which floor) is out of scope but acknowledge that a 2D trail in a multi-story building can mislead the 'already covered' judgment.
- **[MEDIUM] EAS / dev-client build pitfalls specific to this stack are not addressed, and they bite hard mid-project. transistorsoft RNBG is a paid native module requiring a private npm token / postinstall license config; its Expo config plugin must be wired correctly for the foreground-service notification, Info.plist usage strings, and Android 14's new foreground-service-type requirements (FOREGROUND_SERVICE_LOCATION + per-type runtime declaration) — get this wrong and the app crashes on launch on Android 14+ or the service won't start. @rnmapbox/maps also needs a download token at build time and its own config plugin. The risks list correctly notes the license is enforced only in RELEASE builds (so debug 'works', release silently degrades), but the broader trap is that this app can NEVER run in Expo Go and EVERY native-config change requires a full EAS rebuild — slowing iteration, and a wrong plugin order produces opaque native build failures.**
  - Fix: Stand up the EAS dev-client + both native config plugins (RNBG, rnmapbox) in M0/M2 on BOTH platforms and verify a release-channel build on a real device early — don't let the first release build be in M6. Pin the transistorsoft license/token handling in CI secrets from day one and add a smoke test that asserts the SDK reports a valid license in a release build (fail CI otherwise). Explicitly handle Android 14+ foregroundServiceType in the config plugin and test on an Android 14/15 device. Document the 'release-only license' gotcha in the build runbook so a passing debug build is never mistaken for a working release. Verify the iOS Always-permission progressive flow (When-In-Use -> Always upgrade prompt) on a real device, since the simulator lies about this.
- **[MEDIUM] Offline queue ordering and the 'delete-after-200' contract have a correctness edge the design states confidently but doesn't fully cover. The claim: SDK replays ordered batches, ON CONFLICT dedups, client deletes locally only after 200. But: (1) if the Edge Function returns 200 'once GOOD rows persist' while SKIPPING malformed rows (as designed, to avoid infinite retry), the SDK deletes the WHOLE batch including the skipped/dropped rows — those breadcrumbs are now permanently lost, not retried. For a coverage product, silently dropping 'malformed' points (which might just be low-accuracy or odd-but-valid) means real gaps. (2) Partial batch persistence + 200 + ON CONFLICT is fine for dups, but if a row fails for a NON-idempotent reason (e.g. gate rejection), the design's choice to still 200 means permanent loss with no operator visibility. There's no dead-letter / reject-count surfaced anywhere.**
  - Fix: Have the Edge Function return a per-batch result (accepted count, rejected rows + reason) and either (a) write rejected rows to a dead_letter table for inspection, or (b) return a code that tells the SDK to keep them for a bounded number of retries before giving up. Don't conflate 'malformed/unparseable' (drop, but log) with 'rejected by gate' (recoverable once shift state reconciles — keep). Emit a metric/dashboard for rejected-breadcrumb counts so silent loss is observable. Re-examine whether 'skip malformed, return 200' should instead be 'persist what's valid, 200, but report the rejects' so the loss is never invisible.
- **[LOW] Guard map on the mobile device subscribing to its site channel and rendering peers + trails + heatmap is specified the same as the web dashboard, but a phone is far more memory/GPU constrained than a control-room browser. The render-OOM risk (1.4M points) is acknowledged for the dashboard, but the MOBILE client running rnmapbox with multiple ShapeSources, one LineLayer per active guard (the line-gradient workaround), a heatmap layer, AND the background tracking service AND the live socket — on a cheap issued Android phone — is a much tighter budget than the desktop case the math was done for. 'A few hundred guards, one LineLayer each' is fine on desktop GL JS, not on a low-end phone GPU.**
  - Fix: Set a much lower per-guard / per-point cap for the mobile renderer than the dashboard (e.g. only render trails for guards on the same site, shorter default window on mobile, cap concurrent LineLayers). Profile the combined tracking+map memory footprint on a low-end device; tracking reliability must win over map richness if memory pressure forces the OS to kill the app (which would also kill tracking). Consider that the guard mostly needs 'where are my nearby peers' not 'org-wide everything' — scope the mobile map tighter by default.
  - _Missing:_ No 'tracking went dark' detection or alerting anywhere — for a security/coverage product, a silently-stopped tracker is a safety and liability failure, but nothing detects or surfaces it (server-side staleness watchdog + lone-worker-down alert is missing).; No concrete in-app prominent-disclosure / permission priming UX (the pre-OS-prompt consent screen) — only listed as an M6 'asset.' This is both a store-approval requirement and a GDPR consent-quality requirement and needs designing early.; No battery budget, target, or measurement plan on a representative low-end / issued guard phone; battery is the loudest field complaint for tracking apps and is unquantified.; No accuracy/quality filtering of GPS fixes (drop/flag low-accuracy and stale last-known fixes) — basements and urban canyons will inject garbage into the coverage heatmap that the whole product is built to be trusted.; No geofence-assisted auto clock-in/out as a backstop for 'guard forgot to clock in' — a single human error currently produces a total coverage gap in shift-gated mode.; No separation of the long-lived device ingest credential from the 1h interactive access token — the offline buffer flush after a long shift can fail auth as designed.; No handling for what the BEFORE INSERT gate trigger does to legitimately-buffered offline breadcrumbs when clock-in synced late (server-authoritative gate vs offline-first buffer conflict = silent loss).; No Android 14+ foreground-service-type handling called out in the config-plugin work, and no early real-device release-build verification of the transistorsoft license before M6.; No plan for issued-device fleet realities: low-end Android, MDM/kiosk interactions with background services, and devices shared across shifts (whose breadcrumbs are whose at handover?).; No client-side monotonic-clock guard for captured_at to survive OS time/timezone corrections mid-shift, despite captured_at driving ordering, idempotency, and partition routing.
