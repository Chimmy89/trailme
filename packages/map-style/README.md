# @trailme/map-style

One source of truth for how the map looks, so web and mobile render the live markers, per-guard trails, and coverage heatmap with identical colours and identical age-fade. It exports plain JS — never a React component — so it carries no map-library dependency: `guardColor(seed)` (a deterministic FNV-1a hash into a 12-hue palette so the same guard is always the same colour on both platforms), the `TIME_WINDOWS` re-export and `ACCURACY` thresholds, the canonical `SOURCE_IDS`/`FEATURE_PROPS` the GeoJSON must use, the age-fade and heatmap-weight `Expression`s, and layer factories for (a) live guard markers (a `circle` base + a heading `symbol` arrow), (b) a per-guard line-gradient trail that fades newest→oldest, and (c) a monochrome recency-weighted coverage heatmap.

**Byte-portability.** A Mapbox style-spec expression is just a JSON array, and paint/layout values are the same JSON in both libraries, so these objects are byte-portable across mapbox-gl (web) and @rnmapbox/maps (mobile). The only divergence is mechanical: mapbox-gl takes a full layer object via `map.addLayer(spec)`, while rnmapbox takes a flattened `style` prop — spread `{ ...spec.paint, ...spec.layout }` into the layer component. Each platform also registers the marker arrow image itself (`map.addImage` on web, `<Images>` on mobile) under the icon name passed to `liveMarkerArrowLayer`. Trail sources must be created with `lineMetrics: true` (the gradient is driven by `line-progress`), and the window is enforced by trimming LineString coordinates client-side because line-gradient ignores GL filters.
