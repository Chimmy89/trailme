# @trailme/supabase-client

A thin, typed wrapper over `@supabase/supabase-js` that hands back a single `TrailmeClient` (`SupabaseClient<Database>`, with the schema type re-exported from `@trailme/db-types`) so every query in the dashboard, the app, and the edge functions is type-checked against the real database. It exposes three factories: `createAnonClient(env)` for the browser and Expo runtimes (publishable anon key, RLS-bound, realtime on — used by the live per-site channel); `createServiceClient(env)` for trusted server work (service-role key that bypasses RLS, so it carries a runtime guard that throws if a `window` global is present and is meant to be imported only from server-only code); and `createDeviceIngestClient(env, deviceToken)` for code that needs to act as a device using the `~24h` device JWT as a static bearer header. Env (`url`, `key`) is passed in explicitly rather than read from `process.env`, because the dashboard exposes it under `NEXT_PUBLIC_*` and the app under `EXPO_PUBLIC_*` — the caller supplies the right pair. Cookie-bound Next.js SSR clients are intentionally NOT here; they belong in the dashboard app built on `@supabase/ssr` (declared as an optional peer dependency) because they need framework-specific cookie adapters. Note the real durable breadcrumb flush is performed by transistorsoft's native HTTP sync with a static `Authorization` header, not through a supabase-js client at all — `createDeviceIngestClient` is for server/tooling code that needs the same device-token identity.
